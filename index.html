Filecoin is a storage-based cryptocurrency. Rather than mining with computing power, miners provide a data storage service to other users.

Introduction

This whitepaper will cover the entire architecture of the Filecoin system. Beware: this project is still in the design phase, so everything is subject to change.

Please leave any comments or suggestions on the Github issue page.

Filecoin is based on a Bitcoin-like blockchain to keep track of the currency and to index the files being stored by users, as well as a peer-to-peer network used to locate and download files. There are various expected use cases, including cloud storage of personal files (like Dropbox), and highly scalable storage and content delivery (like Amazon S3).

Mining

Overview

All current cryptocurrencies require CPU-based work when generating new blocks. Filecoin uses a similar Hashcash-like algorithm, but instead causes hard drive random access reads to be the bottleneck. The files used in the mining process are owned by other users, so miners are not only securing the network but also providing a file storage service. The amount of storage miners provide linearly scales their expected rewards.

By the nature of this design, Filecoin is ASIC-proof (helping keep mining power decentralized), less power is consumed than CPU mining, and the resources used are not wasted since they directly provide a service.

Inventory Tree

A miner on the Filecoin network maintains a Merkle tree of the files being stored (referred to as the inventory), where all leaves of the tree are pieces of files (referred to as chunks).

All chunks are a uniform length, set by the network. A sufficiently large chunk length should be chosen in order to keep the tree from becoming too large (however, a large chunk size creates larger proofs). A length of 8192 bytes is reasonable, since a miner storing 10 TB of data will only need to use about 1% of capacity to store the tree (using a hash length of 256 bits).

In addition to the inventory tree, a file index is also maintained that specifies the order of the files that make up the inventory.

Algorithm

First, a block header is prepared (which includes information about the block, the Merkle root of the inventory tree and the hash of the inventory index, and a nonce). The miner calculates H(header), where H is the hash function being used by the network.

The hash output is used to specify a Merkle branch in the inventory tree, and the chunk at the end of the branch is retrieved. Only the first m bits of the hash will need to be used, where m is the number of levels in the tree, calculated as ceil(log(n)).

After retrieving the specified chunk from disk, the miner calculates H(hash + chunk), where hash is the hash that was used to specify this chunk.

This process of getting a hash, using it to select a chunk, and hashing it is repeated for a total of k cycles (k is a parameter chosen by the network).

When the final hash is calculated, it is compared to a target value based on the difficulty set by the network (similar to Bitcoin). If the hash is lower than the target, a block hash been found and the miner can broadcast the block, which now includes a verifiable zero-knowledge-proof which consists of k chunks and the Merkle branches proving they are a part of the inventory tree.

Difficulty

When mining Bitcoin, all nodes use the same network-wide difficulty. If that was the method used in Filecoin, mining speed would be based on disk access latency, rather than total amount of chunks stored. That would mean a miner storing 1 GB of data in RAM would mine as many blocks as a miner storing 10 TB of data on hard disks.

To solve this, the target value is derived from not only network difficulty, but is adjusted so that miners storing more chunks are more likely to find blocks. This can be expressed as target = b / difficulty * n, where b is a constant base target value, and n is the number of chunks in the miner's inventory (which can be calculated from the inventory index when verifying).

Attacks

Some attacks may be suggested about this mining system as described, but none that have been presented are viable.

Inventory Spoofing

It would be trivial for a miner to fabricate their inventory index, making it appear that they are storing many chunks (giving them a better difficulty) when they are in reality storing less than this.

However, the miner will need to reject their proof when a chunk is selected that is not being stored, as they cannot provide it. The amount of proofs to be rejected scales exponentially ((n / m) ^ k, where n is the number of actual chunks being stored and m is the number reported), while the difficulty advantage only scales linearly (m). Due to this, it will always be most profitable to use a truthful inventory index.

Hash Grinding

For the first hash in the mining process (the one derived from the block header), many values can be calculated quickly by using different nonces. This makes it possible for a miner who is spoofing their inventory to grind through hashes until they find one that specifies a file they are really storing. This effectively makes the rejected proof rate end up actually being (n / m) ^ (k - 1). Of course, this problem is easy to mitigate by choosing a value of k that is 1 higher than would be sufficient.

Fast Access Mediums

Mining is meant to reward miners who are storing large amounts of data the most, but disk access speed also ends up being a factor. Miners could possibly mine on a small inventory using RAM, generating blocks without providing significant storage service to the network. However, this can be mitigated by enforcing a minimum inventory size and/or not counting the first X chunks into the difficulty advantage. This eliminates miners with very small inventories, which may also help lower the viability of botnet mining.
